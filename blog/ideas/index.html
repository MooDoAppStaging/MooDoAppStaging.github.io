    Moo.do is designed to be flexible from the ground up. Our goal is to allow anyone to organize in a way that makes the most sense for them. If you're interested in how Moo.do can solve different organization problems, we will be posting examples of how different people or groups are using Moo.do.

    To start off I'm going to talk about how we've been using Moo.do to manage its own development. We have spent a majority of the development process working remotely and in vastly different time zones, so keeping everything organized is especially important.

    We found traditional systems for task management and bug tracking to be too heavy. We wasted a lot of our time fighting with software and waiting for things to load, so we designed Moo.do to be a powerful tool for managing a project and tracking its bugs, while remaining very minimal and focused on the content. We use Moo.do to organize everything about itself: features, bugs, ideas, marketing, and feedback. Today I'll focus on how we manage features, bugs, and ideas.

    Organization

    At the beginning of a milestone, we discuss the main things we want to accomplish and lay out a basic plan.

    As we progress through the milestone, we mark tasks complete and add new bugs under Major and Minor Bugs. Our progress is tracked in the Timeline pane where we can see the work each of us is finishing. At the end of a milestone, we move tasks not yet completed into the next release, and discuss what features and bugs from Future we want to bring in. Then we mark the current release complete and move onto the next one.

    Prioritizing and Flagging

    Moo.do has three different priority levels. For us, highest priority means that it should be handled immediately, while lower priorities should be looked at next. We use flagging to highlight things for the other person to look at.

    When I look over what to do for the day, I mark what I'm planning to do as high priority to make them easy to find. Grant prefers to prioritize based on severity, marking the scariest bugs as high priority so he'll keep thinking about them.

    Tagging

    We use tags to assign tasks, group related tasks together, or mark for future action.

    • #GrantAssigned to Grant
    • #JAssigned to Jay
    • #DiscussTo discuss in our next phone call
    • #puntDelay until the next milestone
    • #watchBug that's hard to reproduce - keep an eye out for it

    Searching

    This is where Moo.do is incredibly powerful. Between all the features, bugs, ideas, marketing plans, and user feedback, we have thousands of lines of data. Search lets us focus on what's important right now.

    • To hide irrelevant tasks, we can search for -// -#punt to hide all completed and punted tasks (the search shorthand for 'completed' is //).
    • If I want to see everything assigned to me, I can search for #J
    • If I want to see my todo list for the day, I search for #J! to show only important tasks assigned to me.
    • When we get on the phone we both search for #Discuss and go down the list together.

    Multiple Panes

    It is often quite useful to look at the same data in different ways. A nice way to use search in Moo.do is to have multiple panes open with different searches. Hiding completed and punted tasks shows what's left to do.

    I like to use Moo.do full-screen with three panes open at once:

    • Searched for -// -#punt as shown above. This is where I add, edit, and assign features or bugs. I can add to the current release or scroll down and add them to the next one.
    • Focused on this release and searched for -// -#punt #J to see my tasks.
    • Focused on this release and searched for -// -#punt #J!! which is my todo list for the day.
    These are fully interactive instances of Moo.do. Try your own searches like #Grant to see Grant's tasks.

    Discussion

    In some cases we want to add more data to a task for the other person to work from, or we discover something new that needs to be looked at right away. To add details to a bug or feature, ask questions, add steps to reproduce a bug, or request a code review, we add a sub-item, assign it to the other person using a tag, and flag it to make it stand out. We often have short conversations as we try to reproduce a bug.
